 23   /**
 24   
 25     Invoked by the store whenever it needs to retrieve data matching a 
 26     specific query, triggered by find().  This method is called anytime
 27     you invoke SC.Store#find() with a query or SC.RecordArray#refresh().  You 
 28     should override this method to actually retrieve data from the server 
 29     needed to fulfill the query.  If the query is a remote query, then you 
 30     will also need to provide the contents of the query as well.
 31     
 32     h3. Handling Local Queries
 33     
 34     Most queries you create in your application will be local queries.  Local
 35     queries are populated automatically from whatever data you have in memory.
 36     When your fetch() method is called on a local queries, all you need to do
 37     is load any records that might be matched by the query into memory. 
 38     
 39     The way you choose which queries to fetch is up to you, though usually it
 40     can be something fairly straightforward such as loading all records of a
 41     specified type.
 42     
 43     When you finish loading any data that might be required for your query, 
 44     you should always call SC.Store#dataSourceDidFetchQuery() to put the query 
 45     back into the READY state.  You should call this method even if you choose
 46     not to load any new data into the store in order to notify that the store
 47     that you think it is ready to return results for the query.
 48     
 49     h3. Handling Remote Queries
 50     
 51     Remote queries are special queries whose results will be populated by the
 52     server instead of from memory.  Usually you will only need to use this 
 53     type of query when loading large amounts of data from the server.
 54     
 55     Like Local queries, to fetch a remote query you will need to load any data
 56     you need to fetch from the server and add the records to the store.  Once
 57     you are finished loading this data, however, you must also call
 58     SC.Store#loadQueryResults() to actually set an array of storeKeys that
 59     represent the latest results from the server.  This will implicitly also
 60     call datasSourceDidFetchQuery() so you don't need to call this method 
 61     yourself.
 62     
 63     If you want to support incremental loading from the server for remote 
 64     queries, you can do so by passing a SC.SparseArray instance instead of 
 65     a regular array of storeKeys and then populate the sparse array on demand.
 66     
 67     h3. Handling Errors and Cancelations
 68     
 69     If you encounter an error while trying to fetch the results for a query 
 70     you can call SC.Store#dataSourceDidErrorQuery() instead.  This will put
 71     the query results into an error state.  
 72     
 73     If you had to cancel fetching a query before the results were returned, 
 74     you can instead call SC.Store#dataSourceDidCancelQuery().  This will set 
 75     the query back into the state it was in previously before it started 
 76     loading the query.
 77     
 78     h3. Return Values
 79     
 80     When you return from this method, be sure to return a Boolean.  YES means
 81     you handled the query, NO means you can't handle the query.  When using
 82     a cascading data source, returning NO will mean the next data source will
 83     be asked to fetch the same results as well.
 84     
 85     @param {SC.Store} store the requesting store
 86     @param {SC.Query} query query describing the request
 87     @returns {Boolean} YES if you can handle fetching the query, NO otherwise
 88   */
 89   fetch: function(store, query) {
 90     return NO ; // do not handle anything!
 91   },

 93   /**
 94     Called by the store whenever it needs to load a specific set of store 
 95     keys.  The default implementation will call retrieveRecord() for each
 96     storeKey.  
 97     
 98     You should implement either retrieveRecord() or retrieveRecords() to 
 99     actually fetch the records referenced by the storeKeys .
100     
101     @param {SC.Store} store the requesting store
102     @param {Array} storeKeys
103     @param {Array} ids - optional
104     @returns {Boolean} YES if handled, NO otherwise
105   */
106   retrieveRecords: function(store, storeKeys, ids) {
107     return this._handleEach(store, storeKeys, this.retrieveRecord, ids);  
108   },

110   /**
111     Invoked by the store whenever it has one or more records with pending 
112     changes that need to be sent back to the server.  The store keys will be
113     separated into three categories:
114     
115      - createStoreKeys: records that need to be created on server
116      - updateStoreKeys: existing records that have been modified
117      - destroyStoreKeys: records need to be destroyed on the server
118      
119     If you do not override this method yourself, this method will actually
120     invoke createRecords(), updateRecords(), and destroyRecords() on the 
121     dataSource, passing each array of storeKeys.  You can usually implement
122     those methods instead of overriding this method.
123     
124     However, if your server API can sync multiple changes at once, you may
125     prefer to override this method instead.
126     
127     To support cascading data stores, be sure to return NO if you cannot 
128     handle any of the keys, YES if you can handle all of the keys, or
129     SC.MIXED_STATE if you can handle some of them.
130 
131     @param {SC.Store} store the requesting store
132     @param {Array} createStoreKeys keys to create
133     @param {Array} updateStoreKeys keys to update
134     @param {Array} destroyStoreKeys keys to destroy
135     @param {Hash} params to be passed down to data source. originated
136       from the commitRecords() call on the store
137     @returns {Boolean} YES if data source can handle keys
138   */
139   commitRecords: function(store, createStoreKeys, updateStoreKeys, destroyStoreKeys, params) {
140     var cret, uret, dret;
141     if (createStoreKeys.length>0) {
142       cret = this.createRecords.call(this, store, createStoreKeys, params);
143     }
144         
145     if (updateStoreKeys.length>0) {
146       uret = this.updateRecords.call(this, store, updateStoreKeys, params); 
147     }
148        
149     if (destroyStoreKeys.length>0) {
150       dret = this.destroyRecords.call(this, store, destroyStoreKeys, params);
151     }
152      
153     return ((cret === uret) && (cret === dret)) ? cret : SC.MIXED_STATE;
154   },

157     Invoked by the store whenever it needs to cancel one or more records that
158     are currently in-flight.  If any of the storeKeys match records you are
159     currently acting upon, you should cancel the in-progress operation and

161     
162     If you implement an in-memory data source that immediately services the
163     other requests, then this method will never be called on your data source.
164     
165     To support cascading data stores, be sure to return NO if you cannot

167     SC.MIXED_STATE if you can retrieve some of the.
168     
169     @param {SC.Store} store the requesting store
170     @param {Array} storeKeys array of storeKeys to retrieve
171     @returns {Boolean} YES if data source can handle keys
172   */
173   cancel: function(store, storeKeys) {
174     return NO;
175   },

250   /** @private
251     invokes the named action for each store key.  returns proper value
252   */
253   _handleEach: function(store, storeKeys, action, ids, params) {
254     var len = storeKeys.length, idx, ret, cur, lastArg;
255     if(!ids) ids = [];
256     
257     for(idx=0;idx<len;idx++) {
258       lastArg = ids[idx] ? ids[idx] : params;
259       
260       cur = action.call(this, store, storeKeys[idx], lastArg, params);
261       if (ret === undefined) {
262         ret = cur ;
263       } else if (ret === YES) {
264         ret = (cur === YES) ? YES : SC.MIXED_STATE ;
265       } else if (ret === NO) {
266         ret = (cur === NO) ? NO : SC.MIXED_STATE ;
267       }
268     }
269     return ret ? ret : null ;
270   },
